\section{Experiments}

We evaluate on HumanEval+, MBPP+, LiveCodeBench v6, and a MATH-500-style subset. The primary code metric is pass@1 under hidden tests, while math uses exact-match accuracy on the extracted final answer. The implemented baselines used in the corrected runs are greedy decoding, majority vote over $N$ candidates, a non-adaptive generated-test filter, self-debugging when that path is wired for the benchmark, oracle pass@$N$, and our sequential falsification method. The repository now also includes executable local proxies for S* and CodeT: an adaptive elimination baseline without calibrated ranking, and a non-adaptive agreement-scoring baseline over generated probes. These are useful engineering baselines, but they are still \emph{not} the original external SkyThought or Microsoft implementations, so any results from them should be reported as local proxies rather than as definitive head-to-head comparisons.

\paragraph{Experimental protocol.}
All corrected runs use benchmark-aware local generation and execution, but the serving backend is cluster-specific: earlier DeltaAI runs used a local vLLM-compatible stack, while the clean Anvil rerun path uses the repository's direct \texttt{transformers} backend because recent \texttt{vllm} wheels were incompatible with Anvil's system glibc. For HumanEval+ and MBPP+, we load EvalPlus tasks and evaluate against normalized public and hidden tests \citep{liu2023evalplus}. For LiveCodeBench, we decode both public tests and compressed private tests from the \texttt{code\_generation\_lite} release and execute candidates as standalone programs when the benchmark requires stdin/stdout I/O \citep{jain2024livecodebench}. For MATH-style evaluation, we use a difficulty-balanced subset derived from the Hendrycks test split \citep{hendrycks2021math}. In the corrected falsification path, hidden tests are reserved for final evaluation and never used to choose probes. Whenever possible, runs use three random seeds and report mean accuracy with standard error.

\paragraph{Why the current results mix validation slices and larger corrected runs.}
The original project plan called for a much larger benchmark matrix, but several pipeline issues had to be corrected first: method-aware resume behavior, benchmark-specific prompting for MBPP and LiveCodeBench, and proper stdin/stdout execution for competitive-programming tasks. The results below therefore have two roles. Some are \emph{capped validation slices} that establish end-to-end correctness after these fixes; others are the first \emph{larger corrected runs} launched from that repaired pipeline. Together, they provide materially stronger evidence than unit tests or mock runs, while still stopping short of the final full-benchmark campaign.

\paragraph{Main findings.}
First, falsification delivers the clearest gains on HumanEval+, and these gains persist beyond the tiny validation slices. In the current monitored DeltaAI runs, DeepSeek-R1-Distill-Qwen-7B improves from 32.3\% majority-vote accuracy to 65.2\%, Qwen2.5-Coder-14B improves from $23.8 \pm 1.2$ to $42.4 \pm 1.5$, and DeepSeek-R1-Distill-Qwen-32B improves from $34.5 \pm 0.3$ to $79.6 \pm 1.5$ while the oracle reaches $83.5 \pm 0.6$. These are exactly the kinds of gaps that motivate falsification rather than pure agreement-based selection. The artifact state is slightly ahead of the older prose for two models now. The publication-ready bundle contains a finished 14B single-seed HumanEval+ row at 43.9 for both falsification and self-debug, versus 23.2 for majority vote and 45.7 for the oracle. It also contains a finished 32B single-seed HumanEval+ row at 82.9 for falsification, 82.3 for self-debug, 29.9 for majority vote, and 84.8 for the oracle. We still treat the multi-seed monitoring bundle as the more informative live view, but these completed rows matter because they show the corrected HumanEval path can now produce fully materialized benchmark outputs for the stronger models rather than only progress-derived summaries.

Second, MBPP+ is much more heterogeneous and exposes the hardest scientific objection to the paper. On the completed 378-problem three-seed MBPP+ runs, falsification reaches $33.8 \pm 1.0$ for the 7B model, $75.1 \pm 0.3$ for the 14B model, and $64.4 \pm 1.6$ for the 32B model, while the corresponding self-debug baselines are $33.8 \pm 1.0$, $75.2 \pm 0.4$, and $64.6 \pm 1.5$. This is useful evidence in two directions: falsification is clearly much stronger than greedy or majority vote on the harder models, but the current implementation does \emph{not} yet separate itself cleanly from self-debug on MBPP+.

Third, fixing benchmark semantics matters. Earlier LiveCodeBench debug runs were invalid because the system treated stdin/stdout problems like function-completion tasks. After correcting the loader, prompt format, and execution path, the first larger corrected 7B run reaches 8.0\% on 100 problems, compared with 3.0\% for both greedy and majority-vote selection. The absolute numbers are still low, but the corrected path finally measures the intended task at a more meaningful scale than the earlier 16-problem slice.

Fourth, the larger corrected math run shows that the method is not purely exploiting tiny-slice variance. On a 100-problem three-seed run, falsification improves from $23.0 \pm 2.1$ under majority voting to $33.7 \pm 1.7$, while greedy reaches only $11.7 \pm 2.3$. The oracle is also $33.7 \pm 1.7$ here, indicating that falsification closes the entire observed majority-vote gap on this completed slice. The current math setup remains preliminary because it still uses a subset rather than the full 500 problems, but it is now substantially larger than the original 16-problem pilot.

Fifth, the self-debug comparison is now scientifically central rather than a footnote. On the current monitored HumanEval+ rows, self-debug is 70.1 for the 7B model, 42.4 for the 14B model, and 80.8 for the 32B model, compared with falsification at 65.2, 42.4, and 79.6 respectively. That means the present repository already supports a strong diagnosis of why majority voting fails and how execution-based elimination recovers the oracle gap, but it does not yet support a clean claim that falsification dominates self-debugging as an accuracy method. This is why the paper's strongest current contribution is the combination of benchmark-aware elimination, adaptive-vs-nonadaptive test comparisons, confidence diagnostics, and a cleaner empirical framing of test-time scaling failure on reasoning-oriented code models.

Sixth, broader baseline coverage now makes the paper materially harder to dismiss. The current bundle includes a non-adaptive generated-test filter and executable local CodeT/S* proxies on the rows where those paths have finished. These comparisons are mixed in exactly the way a reviewer would want to see rather than be protected from. On DeepSeek-R1-Distill-Qwen-32B HumanEval+, falsification remains clearly stronger than the non-adaptive generated-test filter (79.6 versus 67.0), which supports the claim that adaptive elimination buys something beyond simply adding more tests. On DeepSeek-R1-Distill-Qwen-7B HumanEval+, however, the local S* proxy currently reaches 76.9 while falsification is at 65.2 and the CodeT proxy is at 69.2, showing that this row is genuinely competitive rather than a one-sided win. Qwen2.5-Coder-14B HumanEval+ is closer: falsification is 42.4, the S* proxy is 41.4, and the CodeT proxy is 39.7. The right reading of these rows is not that the problem has gone away; it is that the repository now exposes a reviewer-grade baseline surface and forces the method to earn its claims against something stronger than majority vote alone. For the same reason, any final main-paper table should distinguish clearly between completed external baselines, local executable proxies, and progress-derived monitoring rows.

\paragraph{Calibration and theory status.}
The repository now generates comparative calibration tables, reliability-diagram payloads, and selective-prediction curves for falsification, majority vote, self-debug, and greedy selection. These artifacts matter because calibration is the clearest contribution that does not collapse into self-debugging. The current capped slices are still too small to support strong final calibration claims, so we do not claim that the confidence proxy is fully calibrated or anytime-valid. Instead, the present empirical result is more modest: the score is useful for within-pool ranking, and the larger scientific question of confidence calibration remains open.

\paragraph{Ablation status.}
The corrected repository now supports the main reviewer-critical sweep families directly in code rather than only in the paper plan: $N$-scaling sweeps, falsification-round sweeps, temperature sweeps, compute-matched comparisons, and component ablations that toggle adaptive probe choice, elimination, and confidence-based survivor ranking. It also now includes explicit probe-family sweeps for mutation, edge-case, random-stress, and differential-only falsification. The clean Anvil rerun queue is still focused on replacing stale headline benchmark numbers first, so these ablations should be read as implemented experimental capacity rather than completed large-scale results at the time of writing. To keep this distinction explicit, the paper bundle now also emits a separate completion table and a baseline-coverage report, which makes it easier to see when a method is absent because it was not run, rather than because it underperformed.

\paragraph{Cluster workflow.}
The current large-run path is hybrid but asymmetric. The best paper-quality results available at writing time come from DeltaAI, where long jobs can be resumed from seed-level progress checkpoints and then converted into reportable payloads even if the final \texttt{results.json} aggregation does not materialize automatically. Anvil remains useful for larger clean rerun campaigns, but those jobs are still more exposed to allocation caps and long dependency chains. Operationally, the two clusters now differ: DeltaAI uses a local OpenAI-compatible serving stack, while the Anvil rerun path uses a project-local conda environment and the repository's direct \texttt{transformers} backend because recent \texttt{vllm} wheels were incompatible with Anvil's system glibc. In both environments, runs checkpoint per-seed progress and then trigger a reporting job. If a long run exits before final aggregation, a progress-collection script reconstructs a valid partial result file so figures and tables remain reproducible. The most recent operational step in this direction is a DeltaAI resume chain for the three incomplete HumanEval seeds in the 7B, 14B, and 32B paper runs.

\paragraph{Limitations.}
This draft is now stronger than a pure pilot paper, but it is still not yet a final large-scale benchmarking paper. The main missing pieces are: (1) completion of the currently resumed DeltaAI HumanEval paper seeds so that the strongest HumanEval rows move from the monitoring bundle into the publication-ready bundle, (2) broader scaling and round-ablation sweeps at the target $N$ values, (3) replacement of the current local S*/CodeT proxies with the original external implementations, (4) fuller LiveCodeBench and math coverage for the larger models, and (5) externally calibrated false-alarm estimates needed for a true anytime-valid confidence guarantee. Most importantly, the current results do not yet answer the reviewer's hardest question in the affirmative: falsification clearly beats majority voting, but it does not yet consistently beat self-debugging or every stronger proxy baseline. Future gains should therefore be measured not against the earlier invalid runs, but against this cleaner and more demanding baseline surface.
